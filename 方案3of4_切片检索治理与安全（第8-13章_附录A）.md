## 8 切片、元数据与版本策略

### 8.1 结构感知切片

摒弃无脑定长切分，按内容类型采用不同切片单元，并保留父级上下文（所属章节/文档标题）作为每个切片的前缀，提升召回可读性。

| 内容类型 | 切片单元 | 附加处理 |
|---|---|---|
| 3GPP 规范 | 条/款，大节内再按语义分块 | 保留章节路径；表格独立成块 |
| TDoc / 主席笔记 | 议程小节 / 文稿章节 | 带 agreed/noted 结论与来源公司 |
| 算法文档 | 小节 + 语义块 | 抽取「算法卡片」字段（用途/输入输出/依赖/对应标准） |
| 代码 | 函数/类等符号级 + docstring | 关联对应设计文档；调用关系入图谱 |
| 仿真数据 | 报告按小节；数值表转结构化 | 指标表单独存储，支持数值型问答 |
| 纪要/邮件/工单 | 按主题/决策项 | 抽「结论/待办/责任人」；脱敏 |

### 8.2 统一元数据 schema

每个切片与 Wiki 条目携带一致的元数据，支撑过滤、路由、权限与治理（字段级 JSON 契约见 §14.4，作为四人协作的强制接口约定）：

- **来源类**：source_type（规范/TDoc/文档/代码/纪要/wiki）、doc_id、URI/路径、章节路径。
- **领域类**：release（Rel-17/18…）、working_group（RAN1–4）、topic（议题/算法名）、standard_ref（关联规范条款）。
- **状态类**：decision（agreed/noted/…）、ambiguous 标记、version、last_updated、time_validity。
- **治理类**：security_level（密级）、owner（责任人）、review_status（草稿/已审核）、confidence。

### 8.3 版本与时效

Release 与文档版本作为一等公民：检索默认锚定用户指定/上下文推断的版本；过期结论保留但降权并标注；GraphRAG 记录「被…取代」关系，支持追溯演进。

## 9 检索与生成管线细节

### 9.1 查询理解与路由

- **意图/实体识别**：抽取议题、Release、算法名、文稿号；判断问题属标准线、算法线还是需二者对齐的混合线。
- **查询改写**：术语归一（内部别名↔标准术语）、多路改写、必要时拆解为子问。
- **路由**：据此选择检索子集与过滤条件（如仅 Rel-18 + 特定工作组），缩小噪声。

### 9.2 混合检索 + 重排

采用业界主流的 **BM25 稀疏召回 + 稠密向量召回 + Reranker 精排** 三段式：BM25 保证编号/术语/型号的精确匹配，向量保证语义召回，Reranker 统一精排；GraphRAG 在此基础上沿关系图补充跨文档关联（如某议题牵连的所有文稿、某算法涉及的所有条款）。Wiki 结论页在精排阶段获得优先加权，确保「有标准答案时先给标准答案」。

### 9.3 生成、引用与防幻觉

- **上下文分层**：Wiki 结论在前（框架）、原文片段在后（证据），显式标注每段来源与版本。
- **逐句引用**：答案关键论断后挂接三级粒度出处；界面可点击跳转原文/规范/代码。
- **无据不答**：上下文不足以支撑时明确说明「未检索到依据」，而非编造；对带 ambiguous 标记的内容主动提示需人工确认。
- **对齐类问答**：输出「算法—标准」对照（如某模块 ↔ TS 条款 ↔ 相关议题结论），这是本库相对通用问答的核心增值。

## 10 知识治理与评估

### 10.1 防腐化机制

- **变更侦测**：每次新会议/新版本/新提交接入，LLM 自动比对既有 Wiki 条目，产出「待复核清单」（被推翻/过时/新增议题）。
- **缺口沉淀**：高频无 Wiki 命中的问题进入「待沉淀」队列，LLM 起草条目候选。
- **时效巡检**：定期扫描 last_updated 超期条目，提醒责任人复核。

### 10.2 审核工作流

LLM 起草 → 领域专家评审（在 Wiki 或 docs-as-code 的 MR 流程中批注）→ 定稿发布 → 变更留痕。自动生成内容默认标「待审核」，未背书内容在答案中降权并标注来源为草稿，防止幻觉被「权威化」。

### 10.3 评估体系

建立金标问答集（覆盖标准线/算法线/对齐类），离线与在线双轨度量：

| 维度 | 指标 | 手段 |
|---|---|---|
| 检索质量 | Recall@k、MRR、命中率 | 金标集 + Ragas/TruLens 离线跑 |
| 答案质量 | 准确率、完整性、忠实度（faithfulness） | LLM 评审 + 专家抽检 |
| 引用质量 | 引用正确率、可溯源率 | 校验引用编号在源库存在且结论一致 |
| 幻觉控制 | 无据编造率、拒答恰当率 | 对抗性/长尾问题集 |
| 体验 | 端到端延迟、用户满意度 | 在线埋点 + 点赞/纠错回流 |

## 11 权限与安全（私有化重点）

- **全内网闭环**：大模型、Embedding、向量库、Wiki、日志全部本地部署，离线可运行，数据与索引不出内网。
- **密级分层**：元数据携带 security_level，检索「先过滤后召回」，按用户角色/密级动态裁剪可见集；高密级算法与客户工单严格隔离。
- **脱敏**：纪要/邮件/工单接入时对个人信息与客户敏感字段脱敏。
- **审计**：检索与问答全程留痕（谁、何时、问了什么、命中哪些源），满足合规追溯。
- **外部素材可信边界**：3GPP 文稿等外来文件解析前做安全处理（如剥离压缩包内异常项），再入库。

## 12 分阶段落地路线图

建议以「先跑通闭环、再扩源提质」的节奏推进，每阶段有独立可验收交付物。周期为参考量级，视团队与算力调整。**v2.0 新增「牵头子任务」列，与第 14 章分工对应。**

| 阶段 | 目标 | 主要交付物 | 周期（参考） | 牵头子任务 |
|---|---|---|---|---|
| **P0 冷启动 / PoC** | 单议题验证嵌套价值：选 1–2 个 3GPP 议题，跑通「原文→时间线→Wiki→问答」 | 可用的单议题问答 demo；评估基线；技术栈拍板 | 2–4 周 | 子任务一牵头，全员参与 |
| **P1 RAG 基座** | 建成解析/切片/元数据/混合检索/推理服务，接入 3GPP 规范+TDoc 与部分算法文档 | 标准线问答上线；引用可溯源；金标评估集 v1 | 4–8 周 | 子任务一（子任务四同步建评估集） |
| **P2 Wiki 层 + 嵌套** | 上线 Wiki 引擎与提炼流水线；议题时间线/算法卡片批量生成 + 审核；Wiki 并入检索池 | 双层嵌套检索；审核工作流；Wiki 优先命中 | 6–10 周 | 子任务二 |
| **P3 治理与评估** | 变更侦测、缺口沉淀、时效巡检、反馈回流；评估体系常态化；GraphRAG 关联 | 防腐化闭环；评估看板；对齐类问答 | 4–6 周 | 子任务四（对齐问答由子任务三牵头） |
| **P4 全源扩展** | 接入代码与仿真数据、纪要/邮件/工单；完善密级隔离；规模化调优 | 四类知识源全覆盖；权限分级；性能达标 | 持续迭代 | 子任务三 / 子任务四 |

## 13 风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| 检索质量不达标（切片/embedding 弱） | 答非所问，信任崩塌 | 结构感知切片 + 混合检索 + Reranker；金标集持续调优 |
| Wiki 腐化过时 | 给出过时「标准答案」误导 | 变更侦测 + 时效巡检 + 强制审核；过期降权标注 |
| 自动生成内容幻觉被权威化 | 错误固化传播 | 草稿默认降权；人工背书才转正；逐句引用校验 |
| 3GPP 版本/章节迁移导致错配 | 跨版本引用错误 | Release 元数据 + 逐会议核对章节；ambiguous 标记透传 |
| 算力不足 | 大模型跑不动/延迟高 | 先用中小模型跑通，MoE/量化 + vLLM 提吞吐；分阶段升级 |
| 密级泄露 | 合规事故 | 先过滤后检索 + 脱敏 + 审计；高密级物理隔离 |
| FTP 抓取被限速/封禁 | 数据接入中断 | 限速 + 断点续传 + 分批；本地缓存镜像 |
| ★四人并行开发接口不一致 | 切片/元数据格式各异，集成时大量返工 | P1 第 2 周冻结元数据 schema v1（见 §14.4）；入库 API 强校验拒收缺字段数据；schema 变更走 MR 评审 |
| ★公共底座成为单点依赖 | 检索基座未就绪时其余三线阻塞 | P0 即发布接口契约与 mock 入库端点，三线可先行开发解析管线；底座部署文档化，任何人可拉起测试环境 |
