## 14 任务拆分与协作计划（四人分工） ★新增

### 14.1 拆分原则

按**知识源维度纵向拆分**：每人端到端负责一条知识线（接入 → 解析切片 → 元数据 → 入库 → Wiki 提炼配合 → 该线问答质量），符合四类知识源「形态迥异、处理策略各不相同」的天然边界（见第 3 章），也便于各自领域专家深耕。

同时，为避免「公共平台无人管」，把四项公共底座能力**分别搭载到与其最相关的知识线上**：

- 检索基座（L4）搭载到 3GPP 线——因为 P0/P1 以标准线跑通闭环，基座与标准线天然同期交付；
- Wiki 层（L3）搭载到算法文档线——算法卡片是 Wiki 层的主要资产形态；
- 知识图谱（GraphRAG）搭载到代码仿真线——代码调用关系与「算法—标准对齐」都依赖图谱；
- 应用层（L5）+ 权限安全 + 评估治理搭载到纪要工单线——该线本身最依赖脱敏与密级管控，且评估治理需要中立视角。

### 14.2 子任务一览

| 子任务 | 知识线 | 兼任公共能力 | 建议人选画像 | 牵头阶段 |
|---|---|---|---|---|
| **一：3GPP 标准线 + 检索基座** | TS/TR 规范、Chairs Notes、TDoc | L1 抓取调度、L4 混合检索/推理服务、检索与生成管线 | 熟悉 3GPP 流程 + 有检索/后端工程能力 | P0、P1 |
| **二：算法文档线 + Wiki 层** | 算法设计/架构/API/SOP 文档 | L3 Wiki 引擎、提炼流水线、审核工作流、嵌套入池 | 熟悉自研算法体系 + 文档治理意识强 | P2 |
| **三：代码仿真线 + 知识图谱** | 实现代码、仿真脚本、性能报告 | GraphRAG 图谱、算法—标准对齐问答 | 熟悉代码库与仿真体系 + 图数据经验 | P3（对齐）、P4 |
| **四：纪要工单线 + 应用/安全/评估** | 会议纪要、邮件、工单 | L5 前端与权限网关、脱敏审计、金标集与评估看板、防腐化运营 | 工程全栈 + 安全合规敏感度 | P1（评估集）、P3 |

### 14.3 各子任务详细说明

#### 子任务一：3GPP 标准知识线 + RAG 检索基座

**目标**：把 3GPP 规范与文稿变成可精确溯源的检索源，并为全组建好统一的检索与生成基座。

**职责范围**

1. **数据接入（L1）**：3GPP FTP 抓取器（限速 ≥2 秒/请求、断点续传、分批调度）、增量同步作业；本地缓存镜像。
2. **解析与切片（L2，3GPP 部分）**：TS/TR 章节感知切片（保留「TS 38.331 §5.3.5」定位）、表格/公式独立成块；复用并产品化现有 parse_chair_notes / build_topic_timeline / download-extract 脚本；Release / working_group / decision / ambiguous 元数据打标。
3. **公共检索基座（L4）**：向量库 + 全文索引 + Reranker + vLLM 推理服务的选型部署；混合检索（BM25 + 向量 + 重排）管线；**统一切片入库 API 与检索服务 API（全组契约，见 §14.4）**；Wiki 优先加权逻辑（与子任务二联调）。
4. **检索与生成管线（第 9 章）第一版**：查询理解/路由、上下文组装、逐句引用、无据不答。

**交付物**：3GPP 规范 + TDoc 全量索引；标准线问答 API（三级引用可点击溯源）；检索/入库服务接口文档；抓取与增量同步作业。

**验收标准**：金标集标准线 Recall@10 ≥ 90%；引用正确率 ≥ 95%；跨 Release 过滤检索可用；接口契约在 P1 第 2 周冻结发布。

**依赖与输出**：无前置依赖（最先启动）；向其余三人输出入库/检索 API 与 mock 端点（P0 末）。

#### 子任务二：自研算法文档知识线 + Wiki 层与审核工作流

**目标**：把高价值但矛盾散乱的算法文档提炼为可信的 Wiki 结论资产，并建成「嵌套」机制本身。

**职责范围**

1. **数据接入与解析（L1/L2，算法文档部分）**：文档库连接器；小节 + 语义块切片；术语归一表（内部别名 ↔ 标准术语，同时供子任务一的查询改写使用）。
2. **算法卡片**：卡片模板设计（用途/输入输出/依赖/对应标准 standard_ref）；LLM 提炼流水线；矛盾与过时版本的消解规则。
3. **Wiki 层（L3）**：Wiki 引擎选型部署（Wiki.js 或 docs-as-code：Markdown + Git + MkDocs）；审核工作流（LLM 起草 → 专家评审 → 背书发布 → 留痕）；3GPP 议题时间线报告（来自子任务一现有管线）批量转 Wiki 条目。
4. **嵌套落地**：Wiki 条目切片、向量化并入检索池（source_type=wiki、携带 review_status）；未审核条目降权规则与子任务一联调。

**交付物**：Wiki 平台上线；算法卡片首批 ≥ 30 条 + 议题时间线条目批量导入；审核 SOP 与责任人矩阵；嵌套检索（Wiki 优先命中）生效。

**验收标准**：Wiki 命中场景答案质量专家评分显著优于纯原文基线；条目审核流转周期 ≤ 3 个工作日；Wiki 高频问题覆盖率 ≥ 70%（P3 复核）。

**依赖与输出**：依赖子任务一的入库契约与检索 API（P0 末可用 mock 先行）；向子任务三输出算法卡片的 standard_ref 字段；向子任务四输出 review_status 元数据。

#### 子任务三：代码与仿真数据知识线 + GraphRAG 知识图谱

**目标**：让代码与仿真数据可检索、可数值问答，并用图谱打通「议题—文稿—算法—条款—代码」关系，交付本库核心增值的对齐类问答。

**职责范围**

1. **代码接入（L1/L2）**：Git 仓库连接器（随提交增量）；函数/类符号级切片 + docstring；代码 ↔ 设计文档关联（与子任务二的算法卡片挂接）。
2. **仿真数据**：测试/性能报告按小节切片；数值表转结构化存储，支持数值型问答（「某场景下吞吐多少」）。
3. **知识图谱（GraphRAG）**：图谱 schema 设计（议题—文稿—算法—规范条款—代码模块，及「被取代」演进关系）；图谱构建流水线（Neo4j / NebulaGraph）；GraphRAG 关联检索接入混合召回（与子任务一联调）。
4. **对齐类问答**：「算法 ↔ TS 条款 ↔ 议题结论」对照表生成与问答模板。

**交付物**：代码/仿真检索上线；知识图谱与关联检索 API；对齐类问答能力；数值型问答能力。

**验收标准**：对齐类金标问题准确率 ≥ 80%；图谱覆盖全部已入库议题与算法卡片；代码检索能定位到函数级。

**依赖与输出**：依赖子任务一的入库契约、子任务二的算法卡片（standard_ref）；图谱 schema P1 期完成设计评审，P2 起构建。

#### 子任务四：纪要/邮件/工单知识线 + 应用层、权限安全与评估治理

**目标**：接入噪声最大、敏感度最高的一线经验知识；同时负责用户直接触达的应用层，以及全库的安全底线与质量标尺。

**职责范围**

1. **数据接入（L1/L2，纪要工单部分）**：纪要/邮件/工单连接器与导出管线；去噪抽取「决策/结论/待办/责任人」；脱敏器（个人信息、客户敏感字段）。
2. **应用层（L5）**：问答前端（对话机器人、Wiki/IDE 内嵌搜索入口）、Agent 编排、引用渲染（三级出处可点击）；点赞/纠错反馈回流通道。
3. **权限与安全（第 11 章）**：security_level 密级 schema 与角色模型；权限网关（先过滤后召回）；审计日志；外部素材入库前安全处理规范。
4. **评估与治理（第 10 章）**：金标问答集建设与维护（标准线/算法线/对齐类，v1 ≥ 200 题）；Ragas/TruLens 离线评估流水线与评估看板；防腐化三机制（变更侦测/缺口沉淀/时效巡检）的运营与「待复核/待沉淀」队列。

**交付物**：问答前端与权限网关上线；脱敏与审计方案通过安全评审；金标集 v1 与周度更新的评估看板；防腐化队列常态运转。

**验收标准**：越权检索事件 0；幻觉率 ≤ 2%；评估看板覆盖 §1.4 全部指标；纪要/工单经脱敏抽检合格率 100%。

**依赖与输出**：金标集与密级 schema 从 P0 即启动（不依赖他人）；前端依赖子任务一的问答 API；向全员输出评估报告与质量红线。

### 14.4 公共接口契约（全组强约束）

四人并行的前提是**契约先行**。以下契约由子任务一在 P0 末发布初版、P1 第 2 周冻结 v1，此后变更须经四人 MR 评审。

**统一切片对象（入库 API 的 payload，示例）**：

```json
{
  "chunk_id": "ts38331-r18-s5.3.5-001",
  "text": "〔TS 38.331 §5.3.5〕RRC connection re-establishment ……（含父级上下文前缀的切片正文）",
  "source_type": "spec | tdoc | doc | code | sim | minutes | wiki",
  "doc_id": "TS 38.331",
  "uri": "repo://specs/38331-i50.docx",
  "section_path": "5.3.5",
  "release": "Rel-18",
  "working_group": "RAN2",
  "topic": "RRC Re-establishment",
  "standard_ref": ["TS 38.331 §5.3.5"],
  "decision": "agreed",
  "ambiguous": false,
  "version": "i50",
  "security_level": "internal",
  "owner": "zhang.san",
  "review_status": "approved",
  "confidence": 0.95,
  "last_updated": "2026-07-01"
}
```

**约定**：

- 所有切片（含 Wiki 条目）经统一入库 API 写入；`source_type`、`doc_id`、`security_level`、`last_updated` 为必填，缺失即拒收。
- `source_type=wiki` 的切片必须携带 `review_status`；`draft` 状态在重排中自动降权并在答案中标注「草稿」。
- 引用渲染以 `doc_id + section_path` 拼装三级出处，各线保证该组合能回链到原文。
- 图谱节点以 `chunk_id` / `doc_id` 为锚点，与向量库、全文索引共用同一 ID 体系。
- 金标集提交格式（问题/标准答案/出处/所属知识线）由子任务四定义，各线每阶段末补充本线题目 ≥ 30 题。

### 14.5 协作时序与机制

**阶段 × 子任务矩阵**（★ 为该阶段牵头）：

| 阶段 | 子任务一（3GPP+基座） | 子任务二（算法+Wiki） | 子任务三（代码+图谱） | 子任务四（纪要+应用/安全/评估） |
|---|---|---|---|---|
| **P0**（2–4 周） | ★ 单议题跑通抓取→解析→检索；发布接口契约与 mock | 单议题时间线 → Wiki 草稿模板 | 图谱 schema 预研 | 金标集框架、密级 schema 预研 |
| **P1**（4–8 周） | ★ RAG 基座 + 标准线问答上线；冻结 schema v1 | 算法文档接入与解析切片 | 图谱 schema 评审定稿、构建启动 | 金标集 v1、评估基线、权限网关设计 |
| **P2**（6–10 周） | Wiki 入池与优先加权联调 | ★ Wiki 层 + 审核工作流 + 嵌套生效 | 图谱与算法卡片挂接、代码接入试点 | 问答前端上线、评估常态化 |
| **P3**（4–6 周） | 检索调优、性能优化 | 防腐化机制配合（待复核处理） | ★ GraphRAG 关联 + 对齐类问答 | ★ 治理闭环 + 评估看板 + 反馈回流 |
| **P4**（持续） | 规模化与延迟达标 | 算法卡片扩量 | ★ 代码/仿真全量接入 | ★ 全源脱敏与密级隔离验收 |

**协作机制**：

- **周例会**（30 分钟）：只同步接口变更、阻塞项与指标趋势，不做技术细节讨论。
- **契约管理**：元数据 schema、入库/检索 API、图谱 schema、金标集格式全部入 Git，变更走 MR、四人评审。
- **阶段末联调周**：每阶段最后一周为集成联调窗口，冻结各自新功能，跑金标集回归。
- **质量红线**：子任务四发布的评估看板为唯一裁判；任一指标跌破 §1.4 目标线，相关线暂停扩源、优先修复。
