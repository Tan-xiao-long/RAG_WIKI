## 4 总体架构

系统自下而上分为五层，数据从原始素材逐层加工为可检索、可信、可服务的知识。

### 4.1 分层架构

| 层 | 职责 | 关键组件 |
|---|---|---|
| **L5 应用与服务层** | 问答机器人 / IDE 与 Wiki 内嵌搜索 / API；查询理解、路由、权限过滤、引用渲染 | 对话前端、Agent 编排、权限网关 |
| **L4 索引与检索层（RAG）** | 混合检索（BM25 + 向量 + 重排）、GraphRAG 关联、上下文组装、生成与防幻觉 | 向量库、全文索引、Reranker、LLM 推理服务 |
| **L3 知识加工层（Wiki）** | LLM 提炼原文为结论页（议题时间线/算法卡片/FAQ/对齐表）；人工审核定稿；版本化 | Wiki 引擎、审核工作流、提炼流水线 |
| **L2 解析与治理层** | 格式转换、结构感知切片、元数据抽取、脱敏、去重、质量校验 | 解析器、切片器、元数据 schema、脱敏器 |
| **L1 数据接入层** | 四类知识源的采集与增量同步（FTP 抓取 / 文件库 / Git / 邮件工单导出） | 3GPP FTP 抓取器、连接器、调度器 |

## 5 数据流与嵌套机制

### 5.1 冷启动：从原文到 Wiki

1. 接入原始素材（规范、TDoc、内部文档、代码、纪要），完成解析、结构切片与元数据打标，建立 RAG 基础索引。
2. LLM 批量提炼：以议题/算法/主题为单位，从原文聚合生成结论页草稿（如 3GPP 议题时间线、算法卡片）。
3. 人工审核：领域专家校对结论、修订术语、背书定稿，发布为 Wiki 正式条目。
4. Wiki 条目同样被切片、向量化，并入检索池——完成「嵌套」。

### 5.2 在线问答：一次查询的旅程

1. **查询理解**：识别意图、抽取议题/Release/算法名等实体，判断路由（标准线 / 算法线 / 混合），必要时改写与拆解。
2. **权限判定**：按用户密级生成检索过滤条件，先过滤后检索。
3. **混合召回**：BM25 全文 + 稠密向量并行召回，GraphRAG 沿「议题—文稿—算法—规范条款」关系补充关联片段。
4. **重排**：Reranker 对候选统一打分；Wiki 结论页享有优先加权，原文片段紧随其后作为证据。
5. **上下文组装**：Wiki 结论在前给框架，原文片段在后给铁证，附带元数据（来源/版本/章节）。
6. **生成与引用**：LLM 基于上下文作答，逐句挂接出处（规范章节 / TDoc 编号 / 文稿小节 / 代码位置）；无据不答。

### 5.3 双向反馈：让 Wiki 不腐化

- **缺口→补页**：检索日志中高频出现、但仅靠原文兜底（无 Wiki 命中）的问题，自动进入「待沉淀」队列，触发 LLM 起草新条目。
- **变更→提示更新**：新会议 Chairs Notes / 新版本规范 / 新代码提交接入后，LLM 比对其与既有 Wiki 条目，标出被推翻或已过时的结论，生成「待复核」提醒给责任人。
- **反馈→修正**：用户对答案的点赞/纠错回流，定位到具体 Wiki 条目或切片，驱动修订与重排权重调整。

## 6 关键技术选型（私有化）

以下为纯内网可离线运行的开源/开放权重栈的推荐组合，均支持商用许可。具体型号请在选型期以当时的评测榜单与自有硬件为准复核——下列为 2026 年年中的主流可选项。

### 6.1 组件推荐

| 组件 | 推荐方案 | 说明 |
|---|---|---|
| **生成大模型** | Qwen3 235B-A22B（MoE，Apache-2.0）；DeepSeek-R1 / V3（MoE，MIT）；Llama 4 Scout（超长上下文，长文档场景） | 中文与技术文本表现好、许可宽松；算力有限时用 Qwen3 32B / Gemma 3 27B 等稠密小模型起步 |
| **Embedding** | BGE-M3（多语言，1024 维，长文友好）；Qwen3-Embedding（强中英跨语） | 支持中英混排与长片段；BGE-M3 原生稠密+稀疏+多向量，契合混合检索 |
| **Reranker** | BGE-Reranker-v2；BCE-Reranker（网易有道，面向 RAG 优化） | 召回后精排，对中文技术问答提升显著 |
| **向量库** | Milvus（大规模）；Qdrant（轻量易运维）；pgvector（已有 PG 时） | 均可私有化；需支持标量过滤（元数据）与混合检索 |
| **全文检索** | Elasticsearch / OpenSearch，或向量库内置 BM25 | 提供 BM25 稀疏召回，与向量互补，对术语/编号精确匹配关键 |
| **知识图谱** | Neo4j / NebulaGraph | 承载 GraphRAG：议题—文稿—算法—条款关系网 |
| **Wiki 引擎** | Wiki.js / Outline（易用），或 docs-as-code：Markdown + Git + MkDocs | docs-as-code 便于版本化、审核走 MR、与代码库同源 |
| **编排 / RAG 框架** | LangGraph / LlamaIndex（可编程）；Dify / RAGFlow（私有化平台，含界面） | RAGFlow 对结构化文档解析友好；要求本地部署 |
| **推理服务** | vLLM / SGLang | 高吞吐本地推理；按并发规模配 GPU（见附录） |
| **离线评估** | Ragas / TruLens（私有化）+ 自建金标集 | 度量召回率、答案正确率、引用正确率、幻觉率 |

> **硬件与规模的取舍**：旗舰 MoE 模型（如 235B/671B）Q4 量化后约需 130–140 GB 显存，适合多卡服务器集中提供推理；若初期算力紧张，用 27B–32B 稠密模型 + 强 Reranker 也能在多数问答场景达到可用质量，先跑通闭环再逐步升级。

## 7 3GPP 专项处理管线（复用现有流程）

3GPP 线可直接嫁接团队已有的「议题时间线」处理流水线，把它作为 Wiki 层 3GPP 条目的自动生成引擎，而非从零另建。现有流程与本架构的映射如下。

| 现有步骤 | 产出 | 在本架构中的角色 |
|---|---|---|
| 解析 Chairs Notes（parse_chair_notes） | records.json：每次会议→议程节→共识文本 + 文稿条目（编号/标题/来源公司/结论/修订链） | 结构化原始知识，进 RAG；结论字段供 Wiki 提炼 |
| 组装议题时间线（build_topic_timeline） | topic_timeline.json：按议题聚合的演进脉络 | Wiki「议题时间线」条目的骨架 |
| 下载/解压 TDoc（download/extract） | 文稿原文 .txt + sections.json（标题→行号） | 支撑最细粒度引用（文稿内小节），原文进 RAG |
| 模型撰写报告（report_template） | 逐议题调研报告（提案人/演进/最终结果/引用） | 直接转为 Wiki 议题条目，人工审核后发布 |

### 7.1 规范（TS/TR）的处理

- **章节感知切片**：按规范目录层级（条/款/表）切片，保留「TS 38.331 §5.3.5」式定位，便于精确引用。
- **版本维度**：同一条款跨 Release 并存，元数据打 Release/版本号；检索默认按目标版本过滤，支持「跨版本 diff」式提问。
- **表格与公式**：单独抽取为结构化片段，避免被定长切分割裂。

### 7.2 引用粒度（与现有三级一致）

- **〔RAN3#121 Chairs Notes §10.2.2〕**——会议 + 议程节；
- **〔R3-234665〕**——文稿编号；
- **〔R3-234665 §2.1〕**——文稿内部章节（依赖下载/解压得到的 sections.json）。

> **接入注意（沿用现有经验）**
>
> - 3GPP FTP 对高频访问敏感：抓取器保持限速（≥2 秒/请求）与断点续传，大批量分批跑。
> - 议程章节号会随工作项阶段迁移（特性期独立章节 → 冻结后并入纠错章节），接入配置需逐会议核对，不可臆测。
> - 修订链交叠、CB 线下产出等情形已在现有解析中标注 ambiguous / origin=(CB)，这些标记应带入元数据，提示答案「存疑需人工确认」。
